# 题库 schema 漂移会直接打坏答题流程

## 现象

形势与政策题库接入后，`多选题` 和 `填空题` 无法正常答题：

- 多选题进入页面后看起来有选项，但原逻辑默认沿用单选判题，导致无法按真实答案集合提交。
- 填空题提交错误答案后，反馈区无法显示正确答案。

## 根因

前端把不同题库当成了“同一业务 shape 的轻微变体”，但实际 schema 已经出现关键漂移：

- sxyz `多选题` 的 `正确答案` 是数组，而旧单选逻辑假设是单值。
- sxyz `填空题` 主要使用 `正确答案` / `答案文本`，而旧组件只读取 `答案`。

问题不在单个按钮或单个组件本身，而在于：**题型运行逻辑和题库字段假设没有一起演进**。

## 试过但没用 / 容易误判的方向

- 只检查“页面有没有渲染出题目和选项”不够，渲染成功不代表判题链路正确。
- 只盯着填空题提交逻辑不够，提交逻辑可能没错，真正坏的是反馈字段读取。
- 把新题库按“数据库题库同类”处理，但不逐题型确认字段契约，会把 bug 藏进运行时。

## 最终有效做法

1. 对新题库先核对 **题型 -> 答案字段 -> 展示字段** 三元契约。
2. 多选题必须有独立渲染/状态/提交分支，不能落入单选默认分支。
3. 反馈组件对答案字段至少做显式兜底：`答案 -> 答案文本 -> 正确答案`，或者更早做统一 schema 适配。

## 以后默认做法

接入新题库或新增题型时，先做一张最小兼容表：

| 题型 | 判题字段 | 反馈字段 | 是否需要独立 UI 状态 |
|---|---|---|---|
| 单选/判断 | `正确答案: string` | `答案文本` | 否 |
| 多选 | `正确答案: string[]` | `答案文本` | 是 |
| 填空 | `正确答案` 或 `答案` | `答案/答案文本/正确答案` | 否 |

如果任一列和现有题库不一致，就不要直接复用旧逻辑。

## 如何更早发现

- 新题库接入当天就抽样检查每种题型至少 1 题真实 JSON。
- 浏览器验收必须覆盖“选中 -> 提交 -> 错误反馈/正确反馈”完整链路，而不只是首屏展示。
- 只要某题型新增了数组答案或字段名变化，就默认需要复核 `App.jsx` 的分支判断和组件反馈字段。

## 一句话结论

**题库接入不是“换 JSON 文件”，而是“重新确认题型契约”；只要 schema 漂移，默认 UI 和判题分支就必须重审。**
